# カテゴリ自動判定ルーター（Phase 0）

`/contents-fullmake テーマ` と引数 1 個で呼ばれた時、テーマ文字列からカテゴリを自動判定する。

## 判定フロー

```
入力: テーマ文字列
  ↓
Step 1: 判定シグナル検出（下記ルール）
  ↓
Step 2: 候補 1位 / 2位を信頼度パーセンテージで算出
  ↓
Step 3: 1位の信頼度で分岐
  ├─ 80% 以上 → 即実行（ユーザーにカテゴリと理由を1行報告）
  └─ 80% 未満 → 候補トップ2をユーザーに提示、選択を待つ
```

---

## カテゴリ判定シグナル

### compare（比較系）

**強シグナル（信頼度 +40 点）**:
- 「vs」「VS」「対」「vs.」を含む
- 「A と B どっち」「AかBか」の構文
- 「比較」「ランキング」「ベスト N」「N 選」
- 2 つ以上の成分名・商品名が並んでいる

**弱シグナル（+15 点）**:
- 「選び方」「違い」
- 「1位」「最強」「最適」

**例**:
- 「ホエイ vs カゼイン」→ compare 95%
- 「プロテイン3000円以下ベスト3」→ compare 90%
- 「EAA と BCAA の違い」→ compare 85%

### ingredient（成分解説系）

**強シグナル（+40 点）**:
- 単一成分名・サプリ名のみ（「アシュワガンダ」「クレアチン」「亜鉛」）
- 「〇〇とは」「〇〇の効果」「〇〇の作用機序」
- 「摂取量」「飲み方」「タイミング」

**弱シグナル（+15 点）**:
- 単一成分名 + 1 ワード（「クレアチン 摂取量」）

**例**:
- 「アシュワガンダ」→ ingredient 90%
- 「クレアチンの摂取タイミング」→ ingredient 85%
- 「マグネシウムの作用機序」→ ingredient 90%

### debunk（誤認訂正系）

**強シグナル（+40 点）**:
- 「嘘」「デマ」「間違い」「間違え」「誤解」
- 「9割が」「ほとんどの人が」
- 「実は」「本当は」「意外と」
- 「無意味」「効果なし」「やめろ」「NG」

**弱シグナル（+15 点）**:
- 「知らない」「気づいていない」

**例**:
- 「プロテインの嘘」→ debunk 95%
- 「9割が間違えるビタミン摂取」→ debunk 90%
- 「実はマルチビタミンは無意味」→ debunk 95%

### entertainment（エンタメ系）

**強シグナル（+40 点）**:
- 実在する有名人・スポーツ選手・芸能人の固有名詞が含まれる（「大谷」「メッシ」「RIZIN」「北島康介」等）
- アニメ・漫画・映画作品名が含まれる
- 「もし〇〇だったら」の仮定構文

**弱シグナル（+15 点）**:
- 「衝撃」「ヤバい」「狂気」

**例**:
- 「大谷の食事メニュー」→ entertainment 85%
- 「鬼滅の刃の呼吸法は筋トレに効くか」→ entertainment 90%
- 「もし筋肉が重力に逆らったら」→ entertainment 80%

### discovery（発見解剖系・10 枚固定）

**強シグナル（+40 点）**:
- 異常な具体数値が含まれる（「1 日卵 16 個」「月の水分 30L」「年間プロテイン 100kg」等）
- 「なぜ〇〇は〇〇できるのか」の分析構文
- 「極限」「限界」「常識外」

**弱シグナル（+15 点）**:
- 驚きの固有事例（「筋肉量 150kg」等）

**例**:
- 「1 日卵 16 個食べる大谷」→ discovery 90%（※ entertainment との複合。discovery 優先）
- 「月の水分摂取 30L の山本義徳」→ discovery 95%

**注意**:
- entertainment と discovery は重なる場合がある。**具体的な数値・極端な事例** があれば discovery 優先。**作品性・キャラクター性** が中心なら entertainment 優先

## 信頼度計算

```
基本点: 50
強シグナル検出: +40（複数検出時は最大のみ加算）
弱シグナル検出: +15（複数検出時は最大のみ加算、強との重複加算なし）
キーワード競合（2カテゴリ以上の強シグナル検出）: 1 位から -20
```

上限 100%、下限 0%。

### 曖昧ケースの典型

| 入力 | 候補1 | 候補2 | 処理 |
|---|---|---|---|
| 「ホエイ」 | ingredient (70%) | compare (50%) | 80% 未満 → ユーザー確認 |
| 「プロテイン」 | ingredient (65%) | compare (55%) | 80% 未満 → ユーザー確認 |
| 「大谷の卵 16 個」 | discovery (85%) | entertainment (70%) | 80% 以上 → 即実行（entertainment を補足報告） |
| 「ビタミン D」 | ingredient (80%) | — | 即実行 |
| 「筋トレサプリのおすすめ」 | compare (75%) | ingredient (45%) | 80% 未満 → ユーザー確認 |

## ユーザー提示フォーマット

### 即実行時（信頼度 80% 以上）

```
テーマ「{theme}」をカテゴリ「{category}」と判定しました。
理由: {主要シグナル}

Phase 1 を開始します...
```

例：
```
テーマ「ホエイ vs カゼイン」をカテゴリ「compare」と判定しました。
理由: 「vs」語を検出（比較系の強シグナル）

Phase 1 を開始します...
```

### 曖昧時（信頼度 80% 未満）

```
テーマ「{theme}」のカテゴリを自動判定しました。候補を選んでください。

1. {候補1カテゴリ}（信頼度 {N}%）
   理由: {シグナル}
2. {候補2カテゴリ}（信頼度 {N}%）
   理由: {シグナル}

番号で選ぶか、別カテゴリ名を入力してください。
（compare / ingredient / entertainment / debunk / discovery のいずれか）
```

例：
```
テーマ「ホエイ」のカテゴリを自動判定しました。候補を選んでください。

1. ingredient（信頼度 70%）
   理由: 単一成分名として検出
2. compare（信頼度 50%）
   理由: ホエイは比較対象として頻出（vs カゼイン等）

番号で選ぶか、別カテゴリ名を入力してください。
```

## 判定エージェント（Phase 0）

オーケストレーター自身が LLM として判定する（エージェント起動不要）。プロンプトは以下：

```
あなたはテーマ文字列からカテゴリを判定するルーター。
references/category-router.md の判定ルールに従い、以下を出力せよ：

入力テーマ: {theme}

出力:
- 候補1: {category} ({N}%)
- 候補1 理由: {シグナル}
- 候補2: {category} ({N}%)
- 候補2 理由: {シグナル}
- 判定アクション: "execute" (候補1が80%以上) or "ask_user" (候補1が80%未満)
```

この出力を受けてオーケストレーターが次の動作を決める。

## 運用ログ

判定の誤り傾向を蓄積するため、以下を Notion の `title_candidates` セクション内に残す（オプション）：

```
自動判定: {判定カテゴリ} ({信頼度}%)
ユーザー最終選択: {採用カテゴリ}
判定一致: yes/no
```

一致率が 70% を下回るようになったら、このファイルの判定ルールをレビュー・調整する。
